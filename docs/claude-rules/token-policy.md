# Token Policy

## 목적
토큰을 아끼는 핵심은 더 적게 말하는 것이 아니라, **덜 읽고 덜 반복하고 덜 출력하는 것**이다.

## 1. 새 작업 시작 정책
- 이전 작업이 끝났고 완전히 다른 작업으로 넘어가면 세션을 정리한 뒤 시작한다.
- 새 작업을 시작할 때는 먼저 작업 범위를 1~2문장으로 다시 고정한다.
- 서로 다른 기능을 한 세션에 길게 이어붙이지 않는다.

## 2. 파일 읽기 정책
- 전체 파일 읽기는 마지막 수단이다.
- 먼저 함수명, 컴포넌트명, 클래스명, 에러 지점, 관련 라인 범위를 특정한다.
- 500줄 이하 파일만 상황에 따라 전체 확인을 고려한다.
- 500~2000줄 파일은 필요한 섹션만 본다.
- 2000줄 이상 파일은 반드시 함수/클래스 단위로 쪼개어 읽는다.

## 3. 반복 읽기 방지 정책
- 같은 파일을 여러 번 처음부터 다시 읽지 않는다.
- 이미 읽은 파일은 변경 구간, diff, 관련 함수 중심으로만 재확인한다.
- 편집 이후 전체 재출력보다 변경 내용 요약을 우선한다.

## 4. 출력 절약 정책
- 테스트 결과는 전체 로그 대신 실패한 테스트명, 에러 원인, 관련 파일만 요약한다.
- git 로그는 최근 필요한 커밋만 제한해서 본다.
- verbose 출력은 기본적으로 금지한다.
- 빌드 로그는 WARNING/ERROR 중심으로만 정리한다.

## 5. 컨텍스트 압축 정책
- 컨텍스트 사용량이 커졌다고 느껴지면 작업 내용을 먼저 짧게 재정리한다.
- 같은 논의를 반복하기 시작하면 요약 후 새 단계로 넘어간다.
- 세션 후반부에는 새 목표 추가보다 현재 작업 마무리를 우선한다.

## 6. 붙여넣기 정책
- 큰 파일을 통째로 붙이지 않는다.
- 필요한 함수, 에러 구간, 설정 블록만 붙인다.
- "이 코드 고쳐줘"보다 "verifyUser 함수에 입력 검증 추가"처럼 좁혀서 전달한다.

## 7. 도구/MCP 정책
- 사용하지 않는 MCP는 연결하지 않는다.
- 긴 도구 설명이나 불필요한 서버 연결은 토큰 낭비로 간주한다.
- 필요한 도구만 켜고, 목적이 끝나면 최소 상태를 유지한다.

## 8. 모델 선택 정책
- 단순 수정, 짧은 디버깅, 파일 정리는 비용 효율 모델을 우선한다.
- 복잡한 설계, 대규모 구조 재편, 깊은 비교 분석일 때만 상위 사고 비용을 감수한다.
- 과한 사고 모드는 기본값이 아니라 예외 상황으로 본다.

## 9. 금지 사항
- 장문 설명으로 사용자 입력을 다시 반복하지 않는다.
- 이미 결정된 내용을 다시 장황하게 논의하지 않는다.
- 전체 로그, 전체 파일, 전체 대화를 무심코 재투입하지 않는다.
- 맥락 없이 "프로젝트 전체를 분석"하는 식으로 시작하지 않는다.
