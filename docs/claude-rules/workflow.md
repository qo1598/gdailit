# Workflow

## 1. 시작 전
- 지금 요청의 목표를 1~2문장으로 고정한다.
- 이 작업이 신규 작업인지, 기존 작업의 연장인지 판단한다.
- 관련 파일 위치를 먼저 찾고, 필요한 파일만 선정한다.
- 범위가 넓으면 더 작은 작업 단위로 쪼갠다.

## 2. 분석 단계
- 관련 파일을 필요한 범위만 읽는다.
- 전체 구조가 필요할 때만 file-map이나 architecture 문서를 먼저 본다.
- 원인 추정은 2~3개 이내로 압축해서 세운다.
- 추정이 좁혀지면 바로 해당 구간을 깊게 본다.

## 3. 수정 단계
- 최소 수정 원칙을 지킨다.
- 동작 유지가 중요한 작업은 구조 개선보다 안정성을 우선한다.
- 하나의 수정에서 여러 문제를 한꺼번에 해결하려 하지 않는다.
- 수정 후 관련 코드만 다시 확인한다.

## 4. 검증 단계
- 전체 테스트보다 변경 영향 범위 테스트를 먼저 수행한다.
- 긴 로그는 수집하지 말고 실패 원인만 확인한다.
- 검증 결과는 통과/실패와 핵심 이유만 보고한다.
- 실패 시 다음 시도 방향을 짧게 제시한다.

## 5. 종료 단계
최종 보고는 아래만 담는다.
- 변경 파일
- 핵심 변경 사항
- 검증 결과
- 남은 리스크 또는 후속 작업

## 6. 작업 중간 압축 규칙
- 세션이 길어지면 현재까지의 결론을 짧게 정리한다.
- 이미 결정된 사실은 반복하지 않는다.
- 중간 요약은 다음 작업자가 바로 이어받을 수 있을 정도로 쓴다.

## 7. 피해야 할 흐름
- 원인도 모른 채 프로젝트 전체 파일을 읽기 시작하는 것
- 수정 전에 장황한 이론 설명부터 늘어놓는 것
- 검증 없이 "끝났다"고 말하는 것
- 작은 버그 수정에 대규모 리팩토링을 섞는 것
