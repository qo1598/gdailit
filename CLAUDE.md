# CLAUDE.md

## 목적
이 프로젝트에서 Claude는 **정확성**을 유지하면서도 **토큰을 절약하는 방식**으로 작업한다.
핵심은 많이 읽고 많이 말하는 것이 아니라, **필요한 범위만 읽고 필요한 결과만 출력하는 것**이다.

## 최우선 운영 원칙
1. 전체 프로젝트를 한꺼번에 읽지 말고, 먼저 **작업 범위**를 1~2문장으로 특정하라.
2. 전체 파일보다 **관련 함수, 컴포넌트, 클래스, 섹션** 단위 확인을 우선하라.
3. 이미 읽은 파일은 반복해서 처음부터 다시 읽지 말고, **변경 구간과 diff 중심**으로 확인하라.
4. 500줄 이상 파일은 전체를 다시 읽지 말고 **필요한 구간만 발췌**해서 보라.
5. 긴 테스트 로그, 빌드 로그, git 로그를 그대로 출력하지 말고 **실패 원인·관련 파일·다음 조치**만 요약하라.
6. 한 번에 여러 목표를 섞지 말고 **하나의 작업 단위**만 처리하라.
7. 출력은 항상 짧고 구조화하라. 장황한 배경 설명보다 **계획 → 실행 → 검증 → 요약** 순서를 지켜라.
8. 컨텍스트가 커지기 전에 스스로 압축적으로 행동하라. 필요한 경우 세션 정리와 요약을 우선하라.
9. 불필요한 MCP, 대규모 검색, 장문 분석을 먼저 하지 말고 **최소 도구, 최소 범위** 원칙을 지켜라.
10. 사용자가 이미 준 정보는 다시 묻지 말고, 주어진 맥락 안에서 바로 작업하라.

## 응답 형식
항상 아래 형식을 기본으로 한다.

### 1) 계획
- 무엇을 할지 2~4줄 이내로 제시

### 2) 실행
- 필요한 파일/구간만 확인
- 수정 또는 분석 수행

### 3) 검증
- 변경 영향 범위 위주로 점검
- 긴 로그 대신 핵심 결과만 보고

### 4) 요약
- 무엇을 바꿨는지
- 왜 바꿨는지
- 남은 리스크가 있는지

## 파일 읽기 규칙
- 전체 파일 참조보다 **정확한 위치 지정**을 우선한다.
- "이 파일 봐"보다는 "auth.js의 verifyUser 함수 확인"처럼 범위를 구체화한다.
- 파일 전체 붙여넣기를 요구받지 않은 이상, 필요한 부분만 활용한다.
- 디버깅 시에도 원인 후보가 보이면 해당 구간만 추적한다.
- 같은 파일을 다시 읽어야 하면 전체 재열람 대신 변경점부터 본다.

## 출력 규칙
- 긴 코드 설명 금지
- 긴 로그 전문 출력 금지
- 필요 이상 대안 나열 금지
- 사용자가 요구하지 않은 이론 설명 금지
- 결론 없는 장문 설명 금지

## 모델/도구 사용 규칙
- 기본적으로 가장 비용 효율적인 방식으로 시작한다.
- 복잡한 설계, 대규모 구조 변경, 깊은 비교 분석이 아니라면 과한 추론/장문 사고를 지양한다.
- 사용하지 않는 기능은 켜 두지 않는다.
- 여러 도구를 동시에 붙이기보다 꼭 필요한 도구만 최소로 사용한다.
